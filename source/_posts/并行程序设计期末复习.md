# 并行程序设计复习







### 概念梳理：

#### 并行计算(处理)：

通常在单一系统中使用两个或多个处理器(计算机)，同时工作以解决单一问题。

#### 分布式计算（处理）：

任何涉及到多台计算机的计算，这些计算机彼此相距甚远，在计算问题或信息处理中各司其职。

#### 并行编程：

人类开发程序的过程，它表达了应该并行执行哪些计算。 

#### 发展趋势：

集成电路制造商的决策是：与其构建更快、 更复杂的单处理器， 不如在单个芯片上放置多个相对简单的处理器。这样的集成电路称为多核处理器。核巳经成为中央处理器或者CPU的代名词。 在这样的设定下， 传统的只有一个CPU的处理器称为单核系统。

#### 冯诺依曼结构：

“经典”的冯·诺依曼结构包括主存、中央处理单元（ Central Processing Unit, CPU ）处理器或核，以及主存和CPU 之间的互连结构

（共享内存，串行执行）

#### 冯·诺依曼瓶颈：

主存和CPU 之间的分离称为冯·诺依曼瓶颈。![image-20200901155449038](%E5%B9%B6%E8%A1%8C%E7%A8%8B%E5%BA%8F%E8%AE%BE%E8%AE%A1%E6%9C%9F%E6%9C%AB%E5%A4%8D%E4%B9%A0/image-20200901155449038.png)

#### CPU:

中央处理单元分为控制单元和算术逻辑单元（ Arithmetic Logic Unit, ALU ）。控制单元负责决定应该执行程序中的哪些指令，而且ALU 负责执行指令。CPU 中的数据和程序执行时的状态信息存储在特殊的快速存储介质中，即寄存器。控制单元有一个特殊的寄存器，叫做程序计数器，用来存放下一条指令的地址。

#### 操作系统：

操作系统（ Operating System , OS ）一种用来管理计算机的软件和硬件资源的主要软件。它决定什么程序能运行以及什么时候运行。它控制运行中程序的内存分配以及对诸如硬盘、网卡等外设的访问。

#### 局部性原理：

有了Cache 后，一个很明显的问题是什么样的数据和指令能够存储在Cache 中。通用的准则基于下面的原理：程序接下来可能会用到的指令和数据与最近访问过的指令和数据在物理上是邻近存放的。局部性原理是指CPU访问[存储器](https://baike.baidu.com/item/存储器/1583185)时，无论是存取指令还是存取数据，所访问的[存储单元](https://baike.baidu.com/item/存储单元/8727749)都趋于聚集在一个较小的连续区域中。

三种不同类型的局部性:

时间局部性（Temporal Locality）：如果一个信息项正在被访问，那么在近期它很可能还会被再次访问。

程序循环、[堆栈](https://baike.baidu.com/item/堆栈)等是产生时间局部性的原因。

空间局部性（Spatial Locality）：在最近的将来将用到的信息很可能与正在使用的信息在空间地址上是临近的。

顺序局部性（Order Locality）：在典型程序中，除转移类指令外，大部分指令是顺序进行的。[顺序执行](https://baike.baidu.com/item/顺序执行)和非顺序执行的比例大致是5:1。此外，对大型[数组](https://baike.baidu.com/item/数组)访问也是顺序的。

指令的顺序执行、数组的连续存放等是产生顺序局部性的原因。

#### cache不一致性:

在写直达（ write-through) Cache 中，当CPU 向Cache 写数据时，高速缓存行会立即写入主存中。在写回（ write-back) Cache 中，数据不是立即更新到主存中，而是将发生数据更新的高速缓存行标记成脏（ dirty ）。当发生高速缓存行替换时，标记为脏的高速缓存行被写入主存中。

直写式缓存方式: 当CPU要将数据写入内存时,除了更新缓冲内存上的数据外,也将数据写在SDRAM中以维持主存与缓冲内存的一致性,当要写入内存的数据多起来的话,速度自然就慢了下来.

![image-20200901160336208](%E5%B9%B6%E8%A1%8C%E7%A8%8B%E5%BA%8F%E8%AE%BE%E8%AE%A1%E6%9C%9F%E6%9C%AB%E5%A4%8D%E4%B9%A0./image-20200901160336208.png)

回写式缓存方式: 当CPU要将数据写入内存时,只会先更新缓冲内存上的数据,随后再让缓冲内存在总线不塞车的时候才把数据写回SDRAM,所以速度自然快得多

#### ILP:

指令级并行：流水线（功能单元分阶段安排）&多发射（多条指令同时启动）

#### TLP:

线程级并行：尝试通过同时执行不同的线程来提高并行性（粗粒度）

#### 细粒度&&粗粒度：

在细粒度（ fine-grained ）多线程中，处理器在每条指令执行完后切换线程，从而跳过被阻塞的线程。尽管这种方法能够避免因为阻塞而导致机器时间的浪费，但它的缺点是，执行很长一段指令的线程在执行每条指令的时候都需要等待。粗粒度多线程为了避免这个问题，只切换那些需要等待较长时间才能完成操作（如从主存中加载）而被阻塞的线程。这种机制的优点是，不需要钱程间的立即切换。但是，处理器还是可能在短阻塞时空闲，线程间的切换也还是会导致延迟。

#### 硬件多钱程

硬件多线程（ hardware multi threading ）为系统提供了一种机制，使得当前执行的任务被阻塞时，系统能够继续其他有用的工作。

#### 同步多线程

同步多线程（ Simultaneous Multithreading, SMT ）是细粒度多线程的变种。它通过允许多个线程同时使用多个功能单元来利用超标量处理器的性能。

#### 阿姆达尔定律:

大致上，除非一个串行程序的执行几乎全部都并行化，否则，不论有多少可以利用的核，通过并行化所产生的加速比都会是受限的。

让s为按顺序完成的工作的比例，所以(1-s)是可并行的比例。P = 处理器数量

加速比(P) = Time(1)/Time(P)

​          <= 1/(s + (1-s)/P) 

​          <= 1/s

即使并行部分完全加速  ，性能仍然受到顺序部分的限制

#### 负载不均衡：

负载不平衡是指系统中的一些处理器由于并行度不够（在该阶段）和任务大小不等而空闲的时间

运算结果依赖于两个或多个事件。

#### 数据依赖：

#### 同步：

用来让并行代码有序控制线程运行，或者有序访问数据。

#### 原子性：

一组操作是原子操作，那么要么一起执行完，要么都不执行，不会出现执行一部分的情况。

读，加，写回必须是原子操作，才能保持程序的正确性

#### 互斥：

任何情况下，最多只能有一个线程可以执行程序代码。



#### 锁竞争和较差的并行粒度。

为了获得互斥锁，至少需要经过几个层次的cache，在寄存器本地copy会出现错误。

互斥锁=1表示锁占用，=0表示空闲。对互斥锁的lock和unlock也不是原子操作。

大多数体系结构都提供了swap或exchange指令，该指令的作用是把**寄存器**和**内存单元**的数据相交换，由于只有一条指令，保证了原子性，即使是多处理器平台，访问内存的总线周期也有先后，一个处理器上的交换指令执行时另一个处理器的交换指令只能等待总线周期。

如何竞争互斥锁呢？“挂起等待”和“唤醒等待”，每个Mutex有一个等待队列，一个线程要在Mutex上挂起等待，首先把自己加入等待队列中，然后置线程状态为睡眠，接着调用调度器函数切换到别的线程。

#### 死锁问题：

一般情况下，如果同一个线程先后两次调用lock，在第二次调用时，由于锁已经被占用，该线程会挂起等待别的线程释放锁，然而锁正是被自己占用着的，该线程又被挂起而没有机会释放锁，因此就永远处于挂起等待状态了，这叫做死锁（Deadlock）。

另一种典型的死锁情形是这样：线程A获得了锁1，线程B获得了锁2，这时线程A调用lock试图获得锁2，结果是需要挂起自己等待线程B释放锁2，而这时线程B也调用lock试图获得锁1，结果是需要挂起自己等待线程A释放锁1，于是线程A和B都永远处于挂起状态了。

写程序时应该尽量避免同时获得多个锁，如果一定有必要这么做，则有一个原则：如果所有线程在需要多个锁时都按相同的先后顺序（常见的是按Mutex变量的地址顺序）获得锁，则不会出现死锁，因为至少还有锁是空闲的，某个线程在获得了这个锁，执行完了之后就会释放自己占有的其他资源（包括锁），于是其他线程就可以顺利推进了。比如一个程序中用到锁1、锁2、锁3，它们所对应的Mutex变量的地址是锁1<锁2<锁3，那么所有线程在需要同时获得2个或3个锁时都应该按锁1、锁2、锁3的顺序获得。如果要为所有的锁确定一个先后顺序比较困难，则应该尽量使用pthread_mutex_trylock调用代替pthread_mutex_lock调用，以免死锁。

#### CPI：

CPI（Clock cycle Per Instruction）表示每条计算机指令执行所需的时钟周期。

CPI通常用于衡量计算机性能，CPI取决于计算机组成和指令系统的结构。CPI=执行程序所需要的时钟周期数/所执行的指令条数。

在计算机架构中，周期、指令（每个指令的时钟周期和指令时钟或 CPI ）是一个用于描述一个处理器性能的一个方面：时钟周期的数量，当一个指令被执行时发生的。它是指令、周期的倒数。

#### 映射:

映射可以是直接的、集合关联的或完全关联的。
直接映射：每个内存块只能映射到一个高速缓存 lineindex = 块地址 % 缓存线数 
集合关联：每个内存块都可以映射到一组缓存线index=块地址%缓存集数
全相联关联：每个内存块可以在任何缓存行中。

#### 解决缓存不一致：

##### 监听式缓存一致性：（分布式的）

核共享一条总线。
总线上传输的任何信号都可以被连接到总线上的所有核 "看到"。
当核0更新存储在其缓存中的x的副本时，它也会在总线上广播这个信息。
如果核1正在 "窥探 "总线，它将看到x已经被更新，它可以将其x的副本标记为无效。

##### 基于字典的协议：（有中央控制）

使用一个叫做字典的数据结构来存储每个缓存行的状态。
当一个变量更新时，字典会被查询，并且在其缓存中拥有该变量的缓存行的核的缓存控制器会失效。

#### 通信障碍：

Cache line存储多个数据时，多个核要去访问cache line得不同数据

可见多个线程操作在同一Cache Line上的不同数据，相互竞争同一Cache Line，导致线程彼此牵制影响，变成了串行程序，降低了并发性。此时我们则需要将共享在多线程间的数据进行隔离，使他们不在同一个Cache Line上，从而提升多线程的性能。

#### 指令级并行（ Instruction -Level parallelism , ILP ）

通过让多个处理器部件或者功能单元同时执行指令来提高处理器的性能。有两种主要方法来实现指令级并行：流水线和多发射。

##### 流水线：

是指将功能单元分阶段安排；

##### 多发射：

多发射是指让多条指令同时启动。这两种方法在现代CPU 中都有使用。是一种单时钟周期内发射多条指令的机制。

实现静态多发射处理器的一种方法，其中决策是在执行前的编译阶段做出的。

实现动态多发射处理器的一种方法，其中决策是由处理器在执行阶段做出的

动态多发射处理器通常也称为超标量处理器。

多发射流水线需要处理下面两个问题：

1）往发射槽( issue slot ）中发射多条指令 ：处理器如何确定在给定的时钟周期发射多少条指令以及发射何种指令呢？

2）处理数据冒险和控制冒险。

超长指令字( Very Long Instruction Word , VLIW)

#### 并行硬件：

Flynn于1972年提出了计算平台的分类法，主要根据指令流和数据流来分类，共分为四种类型。

SISD机器是一种传统的串行计算机，它的硬件不支持任何形式的并行计算，所有的指令都是串行执行。并且在某个时钟周期内，CPU只能处理一个数据流。(intel Pentium 4)

SIMD是采用一个指令流处理多个数据流。这类机器在数字信号处理、图像处理、以及多媒体信息处理等领域非常有效。我们现在用的单核计算机基本上都属于SIMD机器。

MISD是采用多个指令流来处理单个数据流。由于实际情况中，采用多指令流处理多数据流才是更有效的方法，因此MISD只是作为理论模型出现，没有投入到实际应用之中。

MIMD机器可以同时执行多个指令流，这些指令流分别对不同数据流进行操作。最新的多核计算平台就属于MIMD的范畴，例如Intel Core i7 和AMD的双核处理器等都属于MIMD。

#### 两大类并行架构组织

##### 共享内存多处理器架构

1. 连接到内存系统的自主处理器的集合。
2. 支持全局地址空间，每个处理器可以访问每个内存位置。

一个共享内存程序是一个控制线程的集合。
 线程是在程序启动时创建的，也可能是动态创建的。每个线程都有私有变量，例如，本地堆栈变量。 也是一组共享变量，例如，静态变量、共享的公共区块，或全局堆。 线程通过写和读共享变量进行隐式通信。线程通过使用共享变量实现的锁和障碍进行协调。

在共享内存系统中，目前最常用的两种互连网络是总线（ bus ）和交叉开关矩阵（ crossbar ）

顾名思义，交换互连网络使用交换器（ switch ）来控制相互连接设备之间的数据传递。图2-7a是一个交叉开关矩阵（ crossbar ），线表示双向通信链路，方块表示核或者内存模块，圆圈表示交换器。

总线互连：是由一组并行通信线和控制对总线访问的硬件组成的。通信线路有多个设备共享；随着设备数量的增加，性能会降低。

交换互连：，交换互连网络使用交换器( switch) 来控制相互连接设备之间的数据传递。。如交叉开关。

交叉开关矩阵允许在不同设备之间同时进行通信，所以比总线速度快。

但是，交换器和链路带来的开销也相对高。一个小型的基于总线系统比相等规模的基于交叉开关矩阵系统便宜。





##### 分布式内存架构

- 由互连网连接的自主系统的集合。
- 每个系统都有自己独特的地址空间，处理器必须明确通信以共享数据。
- 最常见的例子是通过商品互连连接的个人电脑集群。

一个分布式内存程序由命名的进程组成。

- 进程是一个控制线程加上本地地址空间--没有共享数据。
- 逻辑上共享的数据被分割到本地进程上。
- 进程通过明确的发送/接收对进行通信。
- 每一个通信事件中都隐含着协作。

###### 间接互连

交换机可能不会直接连接到处理器上。

互连的成本可能是一台机器的重要组成部分，受制于

- 开关数量

衡量“同时通信的链路数目”或者“连接性”的一个标准是等分宽度( bisection width)。

想象并行系统被分成两部分，每部分都有一半的处理器或者节点。在这两部份之间能同时发生多少通信呢?

等分带宽(bisection bandwidth) 通常用来衡量网络的质量。

等分带宽不是计算连接两个等分之间的链路数，而是计算链路的带宽

在第一种系统中，每个核访问内存中任何一个区域的时间都互连相同；而在第二种系统中，访问与核直接连接的那块内存区域比访问其他内存区域要快很多，因为访问其他内存区域内存需要通过另一块芯片。因此，第一种系-统称为－致内存访问（ Uniform Memory  一个UMA 多核系统Access, UMA ）系统，而第二种系统称为非一致内存访问（ Nonuniform Memory Access, NUMA)

#### 共享内存编程&& 分布式内存编程区别

共享内存编程

启动单个进程，分叉线程。
线程进行工作。
线程通过共享内存进行通信。
线程通过同步（也是通过共享内存）进行协调。

分布式内存编程

在多个系统上启动多个进程。
进程进行工作。
进程通过消息传递进行通信。
进程通过消息传递或同步（生成消息）进行协调。

#### 动态线程

主线程等待工作，分叉新的线程，当线程完成后，就会终止工作
有效利用资源，但线程的创建和终止很耗时。

#### 静态线程

创建并分配工作的线程池，但在清理前不会终止。
性能较好，但可能浪费系统资源。

#### 线程安全

如果一个函数或库在被多个同时执行的线程调用时 "正确 "运行，那么它就是线程安全的。 
由于多个线程通过共享内存进行通信和协调，所以线程安全的代码会使用适当的同步来修改共享内存的状态。
顺序代码的一些特征可能不是线程安全的

#### PThreads

- PThreads是POSIX标准
- 相对较低的水平
  - 程序员表达线程管理和协调
  - 程序员对并行进行分解，并对计划进行管理。
- 便携式，但速度可能较慢
- 最广泛地用于面向系统的代码，也用于一些种类的应用代码。

#### OpenMP

- OpenMP是较新的标准
- 在共享内存架构上为科学编程提供更高级别的支持。
  - 程序员识别并行性和数据属性，高水平指导调度工作
  - 系统分解并行，管理计划。
- 产生于各种特定架构的实用性。

#### POSIX：

UNIX的便携式操作系统接口。
与操作系统实用程序的接口

#### PThreads: 

POSIX线程接口
系统调用创建和同步线程
在类似UNIX的操作系统平台上应该是相对统一的。
PThreads提供支持：

- 创造并行化
- 同步化
- 没有显式的通信支持，因为共享内存是隐式的；共享数据的指针被传递给线程。



#### 线程安全：

临界区：是更新共享资源的代码块，一次只能由一个线程更新。

Busy-waiting：可以用来避免对关键部分的访问与标志变量和空体的while-loop发生冲突。

mutex: 也可以使用mutex来避免对关键部分的冲突访问。

信号量是避免冲突访问关键部分的第三种方法。它是一个无符号int和两个操作：sem_wait和sem_post。 Semaphores比mutexes更强大，因为它们可以被初始化为任何非负值。

障碍是程序中的一个点，线程在这个点上阻塞，直到所有的线程都到达这个点。

读写锁：当多个线程同时读取一个数据结构是安全的，但如果一个线程需要修改或写入数据结构，那么在修改期间只有该线程可以访问数据结构时，就会使用读写锁。

#### OpenMP 只能够并行化具有典型结构的for 循环。

1、变量index 必须是整型或指针类型(例如，它不能是float 型浮点数)。

2、表达式start 、end 和incr 必须有一个兼容的类型。例如，如果index 是一个指针，那么incr 必须是整型。

3、表达式start 、end 和incr 不能够在循环执行期间改变。

4、在循环执行期间，变量index 只能够被for 语句中的"增量表达式"修改。

这些限制允许运行时系统在循环执行前确定迭代的次数。

(1) OpenMP 编译器不检查被pa r a 1 1 e 1 for 指令井行化的循环所包含的迭代间的依赖关系，而是由程序员来识别这些依赖关系。

(2) 一个或更多个迭代结果依赖于其他迭代的循环，一般不能被OpenMP 正确地并行化。（如 斐波拉契数列）

#### schedule：

schedule子句决定了循环迭代在线程团队中的分配方式。

static([chunk]) 在线程之间静态地划分迭代。

每个线程接收[chunk]迭代，必要时四舍五入以计入所有的迭代。

默认的[chunk]是ceil( #迭代次数/#线程 )

dynamic([chunk])为每个线程分配[chunk]迭代，当一个线程完成时，再分配一个[chunk]迭代。

形成一个逻辑工作队列，由所有循环迭代组成。

默认[块]为1

guided([chunk])动态分配，但[chunk]在每次分配时都会以指数形式减少。

RUNTIME 通过环境变量OMP_SCHEDULE将调度决定推迟到运行时。为这个子句指定一个分块大小是非法的。
AUTO 调度决定被委托给编译器和/或运行时系统。
NO WAIT / nowait: 如果指定了这个子句，那么线程在并行循环结束时不会同步。
ORDERED（有序）：指定线程的迭代是由编译器和/或运行时系统决定的。指定循环的迭代必须像在串行程序中那样执行。

## OpenMP parallel for loops: waiting

When you use a `parallel` region, OpenMP will automatically wait for all threads to finish before execution continues. There is also a synchronization point *after* each `omp for` loop; here no thread will execute `d()` until all threads are done with the loop:

```
a();
#pragma omp parallel
{
    b();
    #pragma omp for
    for (int i = 0; i < 10; ++i) {
        c(i);
    }
    d();
}
z();
```

[![Parallel for loop](%E5%B9%B6%E8%A1%8C%E7%A8%8B%E5%BA%8F%E8%AE%BE%E8%AE%A1%E6%9C%9F%E6%9C%AB%E5%A4%8D%E4%B9%A0./parallel_for_4.png)](http://ppc.cs.aalto.fi/ch3/parallel_for_4.png)

However, if you do not need synchronization after the loop, you can disable it with `nowait`:

```
a();
#pragma omp parallel
{
    b();
    #pragma omp for nowait
    for (int i = 0; i < 10; ++i) {
        c(i);
    }
    d();
}
z();
```

### Interaction with critical sections

If you need a critical section after a loop, note that normally OpenMP will first wait for all threads to finish their loop iterations before letting any of the threads to enter a critical section:

```
a();
#pragma omp parallel
{
    b();
    #pragma omp for
    for (int i = 0; i < 10; ++i) {
        c(i);
    }
    #pragma omp critical
    {
        d();
    }
}
z();
```

[![Parallel for loop and a critical section](%E5%B9%B6%E8%A1%8C%E7%A8%8B%E5%BA%8F%E8%AE%BE%E8%AE%A1%E6%9C%9F%E6%9C%AB%E5%A4%8D%E4%B9%A0./parallel_for_critical.png)](http://ppc.cs.aalto.fi/ch3/parallel_for_critical.png)

You can disable waiting, so that some threads can start doing postprocessing early. This would make sense if, e.g., `d()` updates some global data structure based on what the thread computed in its own part of the parallel for loop:

```
a();
#pragma omp parallel
{
    b();
    #pragma omp for nowait
    for (int i = 0; i < 10; ++i) {
        c(i);
    }
    #pragma omp critical
    {
        d();
    }
}
z();
```

[![Parallel for loop and a critical section](%E5%B9%B6%E8%A1%8C%E7%A8%8B%E5%BA%8F%E8%AE%BE%E8%AE%A1%E6%9C%9F%E6%9C%AB%E5%A4%8D%E4%B9%A0./parallel_for_nowait_critical.png)](http://ppc.cs.aalto.fi/ch3/parallel_for_nowait_critical.png)

### No waiting before a loop

Note that there is no synchronization point *before* the loop starts. If threads reach the for loop at different times, they can start their own part of the work as soon as they are there, without waiting for the other threads:

```
a();
#pragma omp parallel
{
    #pragma omp critical
    {
        b();
    }
    #pragma omp for
    for (int i = 0; i < 10; ++i) {
        c(i);
    }
    d();
}
z();
```

#### MPI: 

在消息传递程序中，运行在一个核－内存对上的程序通常称为一个进程。两个进程可以通过
调用函数来进行通信：一个进程调用发送函数，另一个调用接收函数。我们将使用消息传递的实
现称为消息传递接口（ Message-Passing Interface, MPI ）。MPI 并不是一种新的语言，它定义了一个
可以被C 、C ＋＋和Fortran副程序调用的函数库。
